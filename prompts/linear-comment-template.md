# Linear Comment Template

This file is the **single source of truth** for the comment format posted to Linear by `linear-reporter`. Read this file before building any comment body. Do not deviate from the structure below.

---

## Comment body

The comment body starts directly with the Result line. No title heading, no preamble.

```
**Result:** <PASS | FAIL>  ·  **Env:** <prod|stg>

### Summary
<1–2 sentences. If FAIL, lead with what broke. If PASS, one sentence is enough.>

### Scenarios
- ✅ <scenario title>
- ❌ <scenario title> — <one-line failure reason>

### Notable findings
- **[Product]** <system under test does not match spec/PR diff — product bug or scope-cut>
- **[Warning]** <off-script observation a human should review — ticket↔PR conflict or live-update anomaly>
- **[Spec]** <test setup or scenario design assumes something untrue>
- **[Env]** <environment constraint blocked the run>
- **[Info]** <incidental observation relevant to this run>

### Evidence

**[PASS] Scenario N — <short title>**
<One or two sentences: what the picture proves, not what the picture is.>

![<ticket-lower>-sN-PASS-<slug>](https://uploads.linear.app/...)

**[FAIL] Scenario N — <short title>**
<One or two sentences: what the picture proves.>

![<ticket-lower>-sN-FAIL-<slug>](https://uploads.linear.app/...)

---
*Generated by exploratory-test-agent. Comment only — ticket state unchanged.*
```

---

## Section rules (read before filling in each section)

### Result line
- `PASS` if every primary scenario passed. `FAIL` otherwise. No `MIXED`, `SKIPPED`, or `ERROR`.
- Mention role only when load-bearing (i.e. test ran as `internal` and that matters to interpret the result). Otherwise omit.

### Summary
- 1–2 sentences max.
- FAIL: lead with what broke and (if applicable) which existing ticket it reproduces.
- PASS: one sentence is enough.

### Scenarios
- Every active scenario in the spec gets a line — `✅` or `❌`. No `⏭`, no `not_run`.
- `[skip-on-this-pass]` scenarios are **not listed** — they are explicitly out of scope.
- `❌` lines carry one short failure reason (a clause, not a paragraph). The screenshot carries the rest.
- `✅` lines need no reason — the name is enough.

### Notable findings
- Use the bracketed lead-in tags: `[Product]`, `[Warning]`, `[Spec]`, `[Env]`, `[Info]`.
- Order within the section: Product > Warning > Spec > Env > Info.
- Max 3 bullets. **Omit the section entirely if there is nothing surprising or actionable.**
- Every `live_update_findings` entry in `result.json` becomes a `[Warning]` bullet — copy `title` + `observation` nearly verbatim.
- **Never include:**
  - Future-test speculation ("a follow-up run could test Y")
  - Re-stating scenario verdicts in prose
  - "This is normal" filler
  - Internal-machinery commentary (store names, hook names, etc.)

### Evidence
- One screenshot per active scenario, in spec order.
- Caption format: `**[VERDICT] Scenario N — short title**` then 1–2 sentences on what the picture proves.
- Skip only for non-visual scenarios (pure network-observation); note the skip in the caption.
- Section heading is always `### Evidence` (singular, never `### Screenshots`).
- **Always call `scripts/attach-screenshot-to-comment.py`** — never inline base64 or raw upload calls.

### Footer
- Always end with `---` then the italicised attribution line.
- No run-id, no local file paths, no artifact paths, no `generated.spec.ts` path.

---

## Hard limits

- **≤ 250 words total** (excluding Evidence captions and images).
- **≤ 8 bullets** across Scenarios + Notable findings combined.
- **No emoji** except ✅ and ❌ in the Scenarios list.

---

## What MUST NOT appear anywhere in the comment

- Local file paths (`artifacts/...`, screenshot filenames)
- Run-ids
- Other ticket IDs not already referenced in the source ticket (Linear auto-creates backlinks)
- Credentials or account emails beyond what context requires
- Stack traces, raw JSON, step-by-step execution traces
- The path to `generated.spec.ts` or any portal-archiver branch
